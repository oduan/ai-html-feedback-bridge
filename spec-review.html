MVP 闭环

保存 AI HTML 页面里的用户补充信息，并让下一轮 AI 按 interaction_id 读取。

当前 SPEC 的方向清晰：后端不保存 HTML、不理解页面结构、不定义业务字段，只负责原始补充内容的存取和大小限制。它适合作为 v0.1 的开发依据。

01AI 生成 HTML 页面
02用户补充信息
03页面提交到后端
04按 interaction_id 保存
05下一轮 AI 读取
06AI 继续处理任务

审阅结论

整体合理，适合先做极简后端。CORS、请求体读取限制、保存状态码、时间格式和 Skill 初始目录范围已经确认。

范围边界

v0.1 只做用户补充内容的存取。所有和 HTML 生成、业务理解、版本管理有关的能力都留到后续。

后端要做
  • 接收用户补充内容
  • 按 interaction_id 保存内容
  • 根据 interaction_id 返回内容
  • 限制单次提交大小
  • 保存并恢复 Content-Type
后端不做
  • 不生成或保存 AI HTML 页面
  • 不解析 HTML、业务字段或内容语义
  • 不规定表单结构和提交格式
  • 不做 diff、版本管理、历史记录
  • 不做用户系统、管理后台、权限体系

数据模型

单表 interactions 足够支撑 v0.1。关键点是覆盖保存、原文保存、Content-Type 保真。

字段 类型 约束 语义 审阅备注
interaction_id TEXT PRIMARY KEY 一次 AI HTML 收集用户补充信息的交互实例标识 不是整段对话或登录会话；建议限制长度 1-128，并限制字符集。
content_type TEXT NOT NULL 提交时的 Content-Type 缺省值建议为 text/plain; charset=utf-8。
content TEXT NOT NULL 用户补充给 AI 的原始内容 v0.1 接受文本即可；二进制不在范围内。
created_at DATETIME NOT NULL 首次保存时间，UTC 时间格式 使用 UTC / RFC3339，例如 2026-05-16T09:30:00Z。
updated_at DATETIME NOT NULL 最近覆盖时间，UTC 时间格式 重复提交时更新，不依赖服务器本地时区。
命名决策
  • interaction_id 表示一次具体的 HTML 交互实例。
  • 它不表示整段对话、用户登录会话或筛选流程。
  • 同一个任务或多轮对话中，可以有多个 interaction_id。
  • 后续需要归组时，再增加 task_id、conversation_id 或 turn_id。

HTTP 接口

接口契约简单清楚。实现时要保证读取限制、覆盖行为和 Content-Type 返回一致。

POST 提交用户补充内容
/interactions/{interaction_id}
  • 请求体为任意文本内容
  • 以流式限制方式检查 MAX_CONTENT_SIZE
  • 保存原始请求体和 Content-Type
  • 同一 interaction_id 已存在时覆盖
  • 返回 {"ok": true, "interaction_id": "demo-001"}
GET 获取用户补充内容
/interactions/{interaction_id}
  • 查询指定 interaction_id
  • 存在时返回保存的原始内容
  • 响应头使用保存时的 Content-Type
  • 不存在时返回 404
  • 错误体为 {"ok": false, "error": "interaction not found"}
接口命名决策
  • 不再使用 /interactions/{interaction_id}/submission，因为 interaction_id 已经定位到一次反馈交互。
  • 主接口使用 POST，和 HTML 页面提交表单或脚本提交内容的语义保持一致。
  • 同一个 interaction_id 重复 POST 时，后端执行覆盖保存。
状态码 场景 建议
200 OK读取成功；提交保存成功保存接口统一返回 200，不区分首次保存和覆盖保存。
400 Bad Requestinteraction_id 为空或非法实现前需要固定校验规则。
404 Not Foundinteraction 不存在错误响应使用 JSON。
413 Payload Too Large请求体超过限制必须实现，属于核心边界。
500 Internal Server Error服务端内部错误不要泄露数据库路径或堆栈。

配置与部署

环境变量足够支撑第一版。Compose 需要挂载 SQLite 数据目录，避免容器重建后数据丢失。

变量默认值说明

安全与边界

v0.1 默认适合本地、内网、Demo 或受控环境。公开部署前至少补 token 或其他访问控制。

必须保留
  • 以流式读取方式限制请求体大小
  • 参数化 SQL
  • 不执行用户提交内容
  • 错误信息不泄露内部路径
已确认规则
  • CORS 允许任意来源：Access-Control-Allow-Origin: *
  • 支持 GET、POST 和 OPTIONS
  • 请求体限制在流式读取时执行
  • 保存成功统一返回 200 OK
  • 时间字段使用 UTC / RFC3339
后续增强
  • 简单 token
  • 健康检查：GET /healthz
  • SQLite WAL 和 busy timeout
  • 统一请求日志和错误日志
  • 历史记录
  • Web UI 查看提交
  • 必要时迁移 PostgreSQL

合理性判断
  • 边界足够窄：只保存用户补充内容，避免过早做平台化。
  • 数据模型足够小：单表覆盖保存符合第一版目标。
  • 接口数量合适：两个接口正好对应保存和读取。
  • Go + SQLite 匹配开源 Demo 和本地部署体验。
  • 目录拆分利于 AI 协作，不会把实现都堆在 main.go。
  • CORS、状态码、时间格式和请求体限制已经明确，剩余工程项主要是 healthz、日志、SQLite 参数。

开发优先级与验收

下面这组任务可以直接作为后续开发拆分依据。验收标准保持三句话：能保存、能读取、能拒绝过大内容。

开发顺序
  1. 建立 Go 项目骨架和配置读取
  2. 建立 SQLite 连接和表迁移
  3. 实现 SQLite storage 的保存和读取方法
  4. 实现两个 HTTP handlers
  5. 增加大小限制和错误响应
  6. 补充单元测试和接口测试
  7. 增加 Dockerfile 和 docker-compose
  8. 添加 Skill 初始目录
  9. 增加示例请求和示例 HTML
测试清单
  • 保存 JSON 内容后能原样读取
  • 保存纯文本内容后能原样读取
  • 同一个 interaction_id 重复提交会覆盖旧内容
  • 读取不存在的 interaction_id 返回 404
  • 超过 MAX_CONTENT_SIZE 返回 413
  • 保存和读取时 Content-Type 保持一致
  • 非法 interaction_id 返回 400
  • SQLite 初始化时自动创建表
  • CORS 返回任意来源允许头
  • 保存成功统一返回 200 OK
  • 时间字段使用 UTC / RFC3339
v0.1 完成标准
可以通过 POST 保存某个 interaction_id 的用户补充内容，
可以通过 GET 原样读取该内容，
并且超过大小限制时会被拒绝。
